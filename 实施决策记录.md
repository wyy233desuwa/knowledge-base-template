# 实施决策记录

本文件是知识库的“可复用规则”入口，沉淀跨案例反复适用的实施口径、字段口径、验证约束和历史误判。单次需求的完整过程记录在对应 CASE 文档中；当前功能状态看 [当前结论.md](./当前结论.md)；案例入口看 [案例记录索引.md](./案例记录索引.md)。

## 一、知识库维护规则

### 1. 知识库分层

本知识库建议按三层维护：

- 知识库整理层：`README.md`、`知识库索引.md`、`案例记录索引.md`、`知识库使用说明.md`、`当前结论.md`、`实施决策记录.md`、`历史文档目录.md`。
- 案例记录层：按 [AI新对话启动协议.md](./AI新对话启动协议.md) 的“CASE 命名格式”命名，记录计划、执行、建议、修改、总结。
- 原始资料层：保留历史方案、复验结论和实施总结，用于追溯。

决策：索引只链接次级整理文档，不逐个链接所有原始资料。

### 2. 案例版本与编号规则

CASE 文件名和修改意见文件名统一按 [AI新对话启动协议.md](./AI新对话启动协议.md) 的“CASE 命名格式”执行。

案例写完后，后续针对同一主题的复查建议和修复不走原地修改，而是生成新文件，序号在该主题内递增。

决策：

- 原始计划文档保持不动，保留当时的判断过程。
- 修改建议形成新文件，避免覆盖历史思考。
- 历史误判不要删除，要在总结里标注。
- `案例记录索引.md` 需要同步更新最新版本的入口。

### 3. 案例收口同步规则

单个需求完成后，按信息类型同步到固定入口。

决策：

- 需求过程和事实写入对应 CASE。
- 案例入口、状态和版本关系同步到 [案例记录索引.md](./案例记录索引.md)。
- 当前仍有效的业务口径同步到 [当前结论.md](./当前结论.md)。
- 跨案例可复用的技术规则同步到 [实施决策记录.md](./实施决策记录.md)。
- 知识库结构、字段或维护流程变化同步到 [知识库维护日志.md](./知识库维护日志.md)。
- 需要按阶段追溯的计划和建议，同步到 [历史节点-计划.md](./历史节点-计划.md) 和 [历史节点-建议.md](./历史节点-建议.md)。
- 收口后按 [知识库自检清单.md](./知识库自检清单.md) 复查。

## 二、Bug 处理规则

### 1. Bug 处理必须先计划后修改

遇到 bug、冲突、异常行为或用户指出“这里不对”时，Agent 不能直接修改代码。

决策：

- 先向用户说明当前现象和初步原因。
- 再把 bug 背景、问题判断、涉及文件、准备修改内容、不修改内容、验证方式写进计划书。
- 等用户明确批准后，才允许修改代码。
- 如果修改过程中发现计划外问题，先停下来补充说明，不扩大修改范围。
- Git 冲突也按 bug 流程处理，必须说明左右两边分别代表什么，以及为什么不能直接选一边。

详细流程见 [Bug处理流程.md](./Bug处理流程.md)。
