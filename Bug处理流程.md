# Bug 处理流程

这个文档规定 Agent 遇到 bug、冲突、异常行为或用户指出“这里不对”时的处理方式。

## 核心原则

处理 bug 时，Agent 不能直接改代码。必须先完成三件事：

1. 先告知用户原因。
2. 把 bug 背景、问题判断、准备修改的内容写进计划书。
3. 等用户明确批准后，再开始修改代码。

## 适用场景

- 用户说某个页面、功能、合并结果或代码行为不对。
- 出现 Git 冲突、合并异常、待提交区异常。
- 测试、构建、运行时报错。
- 页面行为和计划、案例、当前结论不一致。
- Agent 发现已有实现可能会破坏用户需求。

## 标准流程

### 1. 先解释原因

Agent 应先用简短文字告诉用户：

- 当前看到的现象是什么。
- 初步判断原因是什么。
- 哪些内容来自用户改动，哪些可能来自远程或历史文档。
- 还需要确认哪些文件或代码。

示例：

```text
这个冲突不是简单选左或选右的问题。你的版本包含某项能力，远程版本包含另一项能力。直接接收任意一边都会丢功能。
```

### 2. 写入计划书

Agent 在修改前必须把本次 bug 的处理计划写进计划书或 CASE 文档。计划至少包含：

- Bug 背景
- 当前现象
- 初步原因
- 涉及文件
- 准备修改内容
- 不准备修改内容
- 验证方式
- 等待用户批准

推荐命名按 [AI新对话启动协议.md](./AI新对话启动协议.md) 的 CASE 命名格式执行，Bug 计划使用 `Bug处理计划` 后缀。

如果 bug 属于已有案例的后续问题，序号递增，不覆盖原计划。

### 3. 等待批准

计划写完后，Agent 必须停止，不继续修改代码，等待用户明确回复批准。

可接受的批准表达：`批准`、`可以改`、`按这个改`、`继续`。

如果用户只是追问原因、补充截图或讨论方案，不视为批准。

### 4. 批准后再修改

用户批准后，Agent 才能执行代码修改。修改时要遵守：

- 只改计划书里列出的文件和范围。
- 如果发现计划外问题，先停下来补充说明，不扩大修改范围。
- 不回滚用户已有改动。

### 5. 修改后总结

修改完成后，需要在对应 CASE 或计划书里补充：

- 实际修改内容
- 与计划是否一致
- 验证命令和结果
- 未处理风险
- 下次 AI 应注意的经验

## 冲突处理特别规则

Git 冲突属于 bug 处理流程的一种。Agent 必须先说明：

- 左侧、右侧分别代表什么。
- 哪些能力必须保留。
- 为什么不能直接接收某一边。
- 准备合并成什么结果。

然后写进计划书，等用户批准后再改冲突文件。

## 优先级

如果本流程和 Agent 默认习惯冲突，以本流程为准。用户当前最新要求仍然最高优先级。
